## Task lifecycle — CLI only, never read or edit the JSON

You MUST NOT read or edit `tasks/*.json` directly. The PRD task JSON is large;
pulling one into context can push you past the model window mid-iteration and
force a retry. Use the `task-mgr` CLI for every task operation instead:

- **Mark a task's status**: emit `<task-status>TASK-ID:done</task-status>`
(statuses: `done`, `failed`, `skipped`, `irrelevant`, `blocked`). The loop
engine parses these and applies them via `task-mgr`.
- **Look up another task**: ALWAYS prefer `task-mgr show <task-id>` (or
`task-mgr list` / `task-mgr next`) — these cover almost every read. Only
as a last resort, if the CLI can't give you what you need, use `jq` to pull
just the field(s) — e.g.
`jq '.tasks[]|select(.id=="FEAT-007")|{id,title,acceptanceCriteria}' tasks/<prd>.json`.
Never `cat`, `Read`, or `grep` the whole file.
- **List tasks / check status**: `task-mgr list`, `task-mgr next`.
- **Add a new task** (review fix / refactor / follow-up): pipe a single task
JSON to `task-mgr add --stdin`. Example:

      echo '{"id":"CODE-FIX-001","title":"Fix race in X","difficulty":"medium","touchesFiles":["src/foo.rs"],"dependsOn":[]}' \\
        | task-mgr add --stdin

Priority is auto-computed; omit for lower priority.
- **Fix in response to a milestone**: pass `--depended-on-by <id>`.
- **Auto-prefix**: loop exports `TASK_MGR_ACTIVE_PREFIX`; bare IDs are
auto-prefixed to the active PRD. Cross-PRD IDs are rejected.
- For anything else (dependencies, status queries, etc.), use the `task-mgr`
CLI — see `task-mgr --help`.

## Completing This Task

This task has no `touchesFiles` — it is a verification, milestone, or polish task.

When all acceptance criteria pass:
1. Commit with message: `feat: SEQ-SNAP-001-completed - [Title]`
If completing multiple tasks: `feat: ID1-completed, ID2-completed - [Title]`
2. Output `<completed>SEQ-SNAP-001</completed>` (using the full task ID shown above).
3. Stop immediately. Do NOT wait on background tasks, `Monitor` streams, polling loops, or `run_in_background` commands after emitting `<completed>`. If any background process is still running, kill it (`KillShell`, SIGTERM) and exit this turn. The loop treats `<completed>` as terminal — anything you wait for afterward is wasted wall-clock until the base timeout kills the subprocess.

The loop will automatically mark the task done and update the PRD.
Do NOT run `task-mgr done` manually.

If you have a strong reason to work on a different eligible task, output `<reorder>TASK-ID</reorder>`.
