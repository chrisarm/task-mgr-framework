### Side-effect taxonomy (cross-cutting)

- **run_tasks bookkeeping**: sites 2, 3, 4, 5, 11, 17 (complete/fail/skip/irrelevant + try_claim + apply auto-claim). Pattern: SELECT EXISTS → INSERT or UPDATE with `ended_at = datetime('now')`. Site 17 uses `MAX(iteration)+1` and `INSERT OR IGNORE`. The TaskLifecycle service must subsume this — current callers thread `run_id: Option<&str>` explicitly.
- **PRD JSON sync**: stderr line `"PRD JSON sync failed for {task}: {err}\n"` is grepped by operators — exact byte-shape is a contract (PRD §6, TEST-INIT-003). Currently triggered from `complete_cmd::complete` and `fail()` paths via the dispatcher, not site-by-site. Sites 13 (claim_slot_task) and 14-21 (recovery + overflow) deliberately skip PRD JSON sync (DB-authoritative-PRD-best-effort invariant).
- **Decay column writes**: sites 3 (`blocked_at_iteration` / `skipped_at_iteration` via COALESCE), 12 (clear both), 18 (`blocked_at_iteration` set). Sites 18 and 3 must not diverge — both use `iteration_counter` from `global_state`.
- **Notes formatting**: sites 1 ([RESET]), 4 ([SKIPPED]), 5 ([IRRELEVANT (learning #N)]), 6 ([UNBLOCKED]), 7 ([UNSKIPPED]), 8/9/10 ([AUTO-* / RESOLVED]), 12 ([DECAY]), 24/25 ([DOCTOR]). All follow `existing\n\n[TAG] reason` pattern via shared `match current_notes` blocks — TaskLifecycle should host one `append_audit_note` helper.
- **consecutive_failures reset**: site 2 only (extra UPDATE after success). Site 18 reads it implicitly via `handle_task_failure`'s `increment_consecutive_failures` (not shown here — separate symbol).
- **last_error clear**: sites 6, 8, 10 (clear on todo transitions); site 18 sets it; site 3 sets it via `error_count`/`last_error` columns.
- **Optimistic-locking pattern**: only sites 11 and 13 (try_claim shape) use rows_affected to detect race; the bulk recovery sites (15, 16) treat 0-row affected as no-op (idempotent).
