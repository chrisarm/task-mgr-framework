## 2026-06-09 - cd8d4641-WIRE-FIX-001

**Wire ExecutionPlan.effort through prompt builders to runner argv**

Root cause: `resolve_execution_plan` correctly computed `ExecutionPlan.effort = models.effort_for(final_provider, difficulty)` (per-provider table), but BOTH prompt builders dropped it and independently recomputed from the static `EFFORT_FOR_DIFFICULTY` table via `effort_for_difficulty()`. Default config matched, so custom per-provider tables (e.g. `providers.grok.effort.high = "medium"`) were silently ignored at the runner argv.

**Fix — carry plan.effort end-to-end (owned `String`, since per-provider levels are runtime config):**
- `PromptResult::cluster_effort`: `Option<&'static str>` → `Option<String>`. Initialized as `plan.effort.or_else(static effort_for_difficulty)` in `prompt/sequential.rs::build_prompt` (static is now only the fallback when the resolved provider has no effort entry).
- `SlotPromptBundle`: added `cluster_effort: Option<String>` field, populated identically from `plan.effort.or_else(static)` in `prompt/slot.rs::build_prompt`. The slot worker now prefers it over re-deriving from `difficulty`.
- `IterationResult::effective_effort` + `SlotEarlyExit::effective_effort` (engine.rs): `Option<&'static str>` → `Option<String>` so the final merged effort flows consistently into the synthetic overflow `PromptResult` and the progress log.
- `iteration_pipeline::ProcessingParams::effective_effort`: `Option<&'static str>` → `Option<&'a str>` (borrowed; only feeds `progress::log_iteration`).

**Dispatch sites now prefer plan effort over the static fallback:**
- Sequential `iteration.rs:459`: `base_effort = prompt_result.cluster_effort` (the plan effort); `effort = overflow_override.or(base_effort)`.
- Wave `slot.rs:150`: `effort = slot.effective_effort(override).or(bundle.cluster_effort)`. The redundant static `effort_for_difficulty` call was removed here (the bundle already bakes the static fallback at build time), dropping the now-unused top-level `model` import.

**Channel discipline preserved:** the prior-overflow effort OVERRIDE channel (`ctx.effort_overrides` → `TaskExecutionPlan::effort` → `SlotContext::effective_effort`) stays `Option<&'static str>` — those are escalation-ladder levels, not config values, and still win over the plan effort. Only the computed/base effort channel became `String`.

**Regression test:** `prompt/sequential.rs::test_cluster_effort_honors_custom_provider_effort_table` — production-shaped grok-primary `ModelsConfig` with `effort.medium = "low"` (vs static medium→"high"); asserts `cluster_effort == Some("low")`. Existing `test_cluster_effort_*` updated to `.as_deref()` comparisons (default claude table still maps medium→"high", unchanged).
