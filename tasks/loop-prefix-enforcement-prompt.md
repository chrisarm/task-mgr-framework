# Claude Code Agent Instructions

You are an autonomous coding agent implementing **Loop Prefix Enforcement** for **task-mgr**.

## Problem Statement

Loop runs that spawn fixup tasks (review/refactor iterations using `task-mgr add --stdin`) intermittently get stuck because the new task's ID and its `dependsOn` references drop the PRD prefix. Concrete failure observed:

```
Warning: <task-status>bae3f9de-REVIEW-001:Done</task-status> dispatch failed:
Cannot complete task 'bae3f9de-REVIEW-001':
unsatisfied dependencies: FIX-PREEXISTING-INTEGRATION-001
```

The dispatch chain matches IDs by literal string, so a bare `FIX-…-001` recorded in `dependsOn` never matches the prefixed row in the DB and the parent task can never complete. Most of the machinery to fix this already exists (`resolve_active_prefix` + `apply_prefix` in `src/commands/add.rs`) but `resolve_active_prefix` returns `Some` only when **exactly one** non-NULL `task_prefix` exists in `prd_metadata`. On a multi-PRD DB it silently returns `None` and the auto-prefix is bypassed.

The fix has three layers, all on the WRITE side:
1. **Loop engine pins `TASK_MGR_ACTIVE_PREFIX`** as an env var on the spawned Claude subprocess (next to the existing `TASK_MGR_DIR` pin).
2. **`resolve_active_prefix` consults the env var first** and hard-errors when it's stale (set but not in `prd_metadata`).
3. **`reject_foreign_prefix`** validates that incoming IDs / `dependsOn` / `--depended-on-by` don't carry a different known PRD's prefix — proactively catches the misfile bug class documented in Learning [2281] and [2236].

Tolerant `<task-status>` dispatch is deliberately out of scope — fixing at the write boundary stops bare IDs from being created in the first place.

## Common Wiring Failures (REVIEW-001 reference)

Most likely misses for this PRD:

- **Env var pinned in SpawnOpts but not actually reaching the child** — verify with a CLAUDE_BINARY wrapper that prints env (Learning [1892], [1624]).
- **A new SpawnOpts call site landed mid-PRD that wasn't updated** — `grep -rn 'SpawnOpts {' src/` and confirm every match has `active_prefix:` set.
- **resolve_active_prefix's signature change leaked an `.ok()` or `.unwrap_or(None)` in the caller** — defeats the hard-error. Caller must use `?`.
- **Env var name string-literal duplicated** instead of using the constant — drift risk.
- **Test pollution** — env-var tests don't restore prior value; later tests see the wrong env state. RAII guard or scopeguard pattern.

// src/commands/add.rs:383
fn resolve_active_prefix(conn: &Connection) -> Option<String> {
    let mut stmt = conn
        .prepare("SELECT task_prefix FROM prd_metadata WHERE task_prefix IS NOT NULL")
        .ok()?;
    let prefixes: Vec<String> = stmt
        .query_map([], |row| row.get(0))
        .ok()?
        .filter_map(|r| r.ok())
        .collect();
    if prefixes.len() == 1 {
        Some(prefixes.into_iter().next().unwrap())
    } else {
        None
    }
}

### Env-var test guard pattern (mitigates Learning [1685])

```rust
// Tiny RAII restore — drop it into add.rs::tests
struct EnvVarGuard {
    name: &'static str,
    prior: Option<String>,
}
impl EnvVarGuard {
    fn set(name: &'static str, value: &str) -> Self {
        let prior = std::env::var(name).ok();
        std::env::set_var(name, value);
        Self { name, prior }
    }
    fn unset(name: &'static str) -> Self {
        let prior = std::env::var(name).ok();
        std::env::remove_var(name);
        Self { name, prior }
    }
}
impl Drop for EnvVarGuard {
    fn drop(&mut self) {
        match &self.prior {
            Some(v) => std::env::set_var(self.name, v),
            None => std::env::remove_var(self.name),
        }
    }
}
```
