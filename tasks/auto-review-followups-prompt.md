## Key Learnings (from task-mgr recall)

- **[2419]** Slash-command argv re-tokenizes on whitespace inside the body — Claude's slash-command parser splits the single argv body on whitespace, so a PRD path with a space silently fragments. This is the root cause of FEAT-001's bug. Defense: suppress at the launch boundary, do NOT gamble on parser quoting.
- **[2394]** Shared gateway functions enforce consistent behavior across call sites — `auto_review::maybe_fire` is the single gateway used by both `main.rs::Commands::Loop` and `batch.rs::run_batch`; any new gate added to it automatically applies to both paths. FEAT-001's guard belongs here, not duplicated at the two call sites.
- **[2411]** Return value unused prevents exit code side effects — `maybe_fire` is called without binding its return at both call sites; this is the load-bearing invariant for the exit-code contract. Do NOT introduce a path where the return is consumed or unwrapped.
- **[2402]** Guard-chain functions (71+ lines) are acceptable if structured as flat returns — `maybe_fire` is already a flat-return guard chain; the new whitespace guard fits the same pattern. Don't extract a helper for a one-shot.
- **[2387]** cargo test requires `--` separator before module-path filters — `cargo test -p task-mgr loop_engine::auto_review` works; `cargo test -p task-mgr -- loop_engine::auto_review` is the alternate form.
- **[2370]** Derive Default for struct-initialization refactoring — FEAT-003's chain_base hoist is the smaller-scope variant of this pattern (subexpression deduplication vs. struct restructuring). Same DRY motivation.
- **[1409]** Default trait spread in tests insulates against new struct fields — relevant for FEAT-003's regression: existing batch.rs tests that use `..Default::default()` continue to compile after refactor.
- **[528]** `Option<String>` field requires `.clone()` when borrowed via reference — applies to the chain_base hoist: closure must `.clone()` the snapshot, can't move.
- **[2384]** ProjectConfig serde deserialization with multiple field defaults works — unrelated to this PRD's tasks but confirms the FEAT-001/002 changes compose cleanly with the autoReview/autoReviewMinTasks config layer.
- **[2391]** Pure helper functions enable unit testing of selection logic — `pick_review_target` and `should_fire` are both pure; FEAT-002's zero-threshold test slots into the existing test module without scaffolding.

### From `## Auto-launch /review-loop after loop end`

- Default behavior: fires when `autoReview: true` (default) AND `tasks_completed >= autoReviewMinTasks` (default 3).
- CLI overrides (clap-enforced mutual exclusion): `--auto-review` (force on; threshold 1) / `--no-auto-review` (force off).
- Batch mode: ONE review fires at end-of-batch for the LAST successful PRD meeting threshold — never per-PRD.
- Suppression cases (prints recovery hint, continues, exit code unchanged):
  - Non-TTY stdout (CI, pipes).
  - `tasks/<prd>.md` not found AND `tasks/prd-<stem>.md` not found.
  - Worktree path missing or cleaned up.
- Process model: `Command::status()` — blocking spawn, stdin/stdout/stderr inherit so the review session is interactive. `ANTHROPIC_API_KEY` and other env vars inherit automatically.
- Module: `src/loop_engine/auto_review.rs` — `Decision`, `resolve_decision`, `should_fire`, `ReviewLauncher` trait, `maybe_fire`.
- **Invariant**: auto-review failure NEVER changes the loop or batch exit code.
- **Known footgun — paths with whitespace** (added 2026-05-12 via /compound): `ProcessLauncher::launch` interpolates the PRD path into a single slash-command argv element; Claude re-tokenizes on whitespace so a path with spaces fails to find the file. Project convention is space-free `tasks/<feature>.md` paths; FEAT-001 adds a runtime guard for defense in depth.
