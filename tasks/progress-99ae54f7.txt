### Change 2: Step 4 — Document prefix behavior

After the list of prefixes (after `POLISH-xxx` line), **add**:

```markdown
#### Task ID Prefix Behavior

When `task-mgr init` imports a JSON file, task IDs are prefixed with an 8-char hex string
derived from `md5(branchName + ":" + filename)[..8]`, e.g. `99ae54f7-FEAT-001`.

The prefix is resolved in this order:
1. `taskPrefix` field in the PRD JSON (if present — pinned, stable across re-imports)
2. Auto-generated: `md5(branchName + ":" + filename)[..8]` (written back to JSON on first import)

Include `"taskPrefix"` in your PRD JSON to pin the prefix; omit it to auto-generate:
```json
{
  "version": "1.0",
  "taskPrefix": "99ae54f7",
  "branchName": "feat/my-feature",
  ...
}
```

The auto-generated prefix is written back to the JSON file after the first import, ensuring
stable IDs across re-imports. Use the prefix as a namespace when running multiple PRDs in the
same database (e.g., batch mode).
```

**After:**

  "description": "Address finding from CODE-REVIEW-1: [details]",
  "rootCause": "[Underlying cause — helps agent understand what went wrong, not just what to change]",
  "exactFix": "[Precise change needed — specific function, line range, or pattern to apply]",
  "acceptanceCriteria": ["Issue resolved", "No new warnings"],
  "priority": 14,
  "passes": false,
  "taskType": "implementation",
  "maxRetries": 2,
  "dependsOn": [],
  "touchesFiles": ["affected/file.rs"]
}
```

Note: `maxRetries: 2` on CODE-FIX tasks enables auto-block if the fix repeatedly fails — prevents
the loop from wasting iterations on an unfixable issue.
