# Claude Code Agent Instructions

You are an autonomous coding agent implementing **Unify sequential and parallel-slot execution paths** for **task-mgr**.

## Problem Statement

`run_iteration` (sequential) and `run_slot_iteration` + `process_slot_result` (parallel-wave) have drifted into two materially different pipelines. Wave mode silently skips behaviors the sequential path treats as core: slot prompts ship without learnings injected, slot output is never extracted for new learnings, and bandit feedback never updates. As parallel waves become the default for multi-task PRDs, this gap is making the loop progressively dumber the more it is used in its primary mode.

This PRD unifies prompt construction (three-builder model: `core` + `sequential` + `slot`) and post-Claude processing (single `iteration_pipeline` module). It also folds in per-task crash tracking, the wave-mode "already complete" fallback, and per-slot PromptTooLong recovery.

## Non-Negotiable Process (Read Every Iteration)

Before writing code:

1. **Internalize quality targets** — Read `qualityDimensions`; that's what "done well" means for THIS task.
2. **Plan edge-case handling** — For each `edgeCases` / `invariants` / `failureModes` entry on the task, decide how it'll be handled before coding.
3. **Pick an approach** — State assumptions in your head. Only for `estimatedEffort: "high"` or `modifiesBehavior: true` tasks, name the one alternative you rejected and why.

After writing code, the scoped quality gate is your critic — run it (Quality Checks § Per-iteration). Don't add a separate self-critique step; the linters, type-checker, and targeted tests catch more than a re-read does.

## Priority Philosophy

In order: **PLAN** (anticipate edge cases) → **PHASE 2 FOUNDATION** (~1 day now to save ~2+ weeks later — take it, we're pre-launch) → **FUNCTIONING CODE** (pragmatic, reliable) → **CORRECTNESS** (compiles, type-checks, scoped tests pass deterministically) → **CODE QUALITY** (clean, no warnings) → **POLISH** (docs, formatting).

Non-negotiables: tests drive implementation; satisfy every `qualityDimensions` entry; handle `Option`/`Result` explicitly (no `unwrap()` in production). For `estimatedEffort: "high"` or `modifiesBehavior: true` tasks, note the one alternative you rejected and why. For everything else, pick and go.

**Prohibited outcomes:**

- Tests that only assert 'no crash' or check type without verifying content
- Tests that mirror implementation internals (break when refactoring)
- Abstractions with only one concrete use
- Error messages that don't identify what went wrong
- Catch-all error handlers that swallow context
- String-sniffing for outcome dispatch where a structured discriminator is available (per learning #2005)
- Reading `&Connection` on a slot worker thread — every DB read MUST stay on the main thread (rusqlite is `!Send`, learnings #1893 / #1852 / #1871)
- Adding a non-Send field to `SlotPromptBundle` (e.g., `Rc`, `RefCell`) — guarded by compile-time assert
- Behavior changes in Phase A — the prompt-builder split MUST be byte-identical for the snapshot fixture

## Quality Checks

The full test suite is expensive. Per-iteration tasks run a **scoped** gate; **milestones** run the full gate.

### Per-iteration scoped gate (implementation / test / fix tasks)

Format → type-check → lint → **scoped tests for touched files** → pre-commit hooks. Fix every failure before committing.

```bash
cargo fmt --check
cargo check
cargo clippy -- -D warnings
cargo test -p task-mgr <module_or_fn_name>          # narrower match within the crate
cargo test 2>&1 | tee /tmp/test-results.txt | tail -3 && grep "FAILED\|error\[" /tmp/test-results.txt | head -10
```

Scoping heuristic: start from `touchesFiles`. For each file, run `cargo test` with a name filter that matches the affected module's tests. **Do NOT** run the entire workspace test suite without a filter during regular iterations — that's the milestone's job.

### Milestone gate (MILESTONE-1 / MILESTONE-2 / MILESTONE-FINAL)

Milestones run the **full, unscoped** suite on a clean checkout and must finish green:

```bash
cargo fmt --check && cargo check && cargo clippy --all-targets -- -D warnings && cargo test 2>&1 | tee /tmp/test-results.txt | tail -3 && grep "FAILED\|error\[" /tmp/test-results.txt | head -10
```

If ANY test fails — including pre-existing failures — the milestone fixes them or spawns FIX-xxx tasks. Default: **attempt every failure** inline. Pragmatic escape hatch only above ~12 unrelated failures (see CLAUDE.md milestone protocol).

## CLAUDE.md Excerpts (only what applies to this PRD)

These bullets were extracted from `CLAUDE.md` for the subsystems this PRD touches. **Do NOT Read the full file.**

### From "Slot merge-back conflict resolution"
- `merge_slot_branches_with_resolver` (`src/loop_engine/worktree.rs`) runs `git merge --no-edit` from slot 0 for each ephemeral slot branch.
- `ClaudeMergeResolver` (`src/loop_engine/merge_resolver.rs`) is invoked on conflict. The resolver's `Resolved` claim is **never trusted** — caller re-inspects MERGE_HEAD and HEAD post-spawn and downgrades a lying resolver to `failed_slots` with a forced `git reset --hard pre_merge_head`.
- `SlotFailureKind::ResolverAttempted` vs `PreResolver` lets engine.rs pick the right warning text without string-sniffing.
- **Implication for this PRD**: Wrapper commit and external-git reconciliation stay at the run_loop / run_wave_iteration boundary — pipeline does NOT touch git.

### From "Learning Creation Chokepoint"
- All production code paths that create learnings must go through `LearningWriter` in `src/learnings/crud/writer.rs`. Ensures every new learning gets an Ollama embedding scheduled (best-effort, graceful when down).
- `extract_learnings_from_output()` in `src/learnings/ingestion/mod.rs` is one of the production paths using `LearningWriter` — pipeline calls this; no rewrite needed.
- `LearningWriter::flush(conn)` is called **after** any enclosing transaction has committed — this is where the Ollama HTTP call happens. Never flush inside a `rusqlite::Transaction`.

### From "Learning Supersession"
- Use `task-mgr learn --supersedes <old-id> ...` or `task-mgr edit-learning <new-id> --supersedes <old-id>` to mark superseded learnings.
- `task-mgr recall` excludes superseded learnings by default. Pass `--include-superseded` to see them.
- **Implication for this PRD**: When shipping FEAT-007, supersede #106 / #1813 / #213. When shipping FEAT-008, supersede #2029. Done in VERIFY-001.

### From "Overflow recovery and diagnostics"
- The four-rung recovery ladder lives at `overflow::handle_prompt_too_long` in `src/loop_engine/overflow.rs`.
- **Order of operations is contractual** (do NOT reorder): ctx update → DB UPDATE → stderr → dump → JSONL → rotate. Recovery state must be durable before any best-effort observability writes.
- Banner annotation gates on `IterationContext::overflow_recovered` (a `HashSet<String>`), NOT on `model_overrides` — see learning #893.
- The original model is captured first-overflow only via `IterationContext::overflow_original_model.entry().or_insert_with(...)`.
- **Implication for this PRD**: FEAT-008 wires per-slot dispatch but does NOT change the order of operations or the banner-annotation contract.

### From "task-mgr Workflow Patterns"
- `task-mgr init --from-json` (bare, no `--append`) is destructive. NEVER use during a live loop. Correct incremental sync: `task-mgr init --from-json tasks/<prd>.json --append --update-existing`.
- For machine-readable state on a task: embed in JSON entry and re-sync via `--append --update-existing`. For cross-task knowledge: `task-mgr learn`.
- `task-mgr` CLI has no "comment" or "notes-edit" subcommand.

### From "Soft-dep guard for milestone scheduling"
- `build_scored_candidates` defers candidates whose acceptance criteria reference an active spawned-fixup prefix (`REFACTOR-N`, `CODE-FIX`, `WIRE-FIX`, `IMPL-FIX`).

## Data Flow Contracts

These are **verified access patterns** for cross-module data structures. Use these exactly — do NOT guess key types from variable names or comments.

### Main thread builds bundle → ships to worker → worker emits output → main thread post-processes

```rust
// Main thread, before spawning a slot worker:
let bundle = prompt::slot::build_prompt(&conn, &task, &params); // SlotPromptBundle (Send)
let ctx = SlotContext {
    slot_index,
    working_root,
    prompt_bundle: bundle, // typed field, Send — moves into the worker
    last_activity_epoch,
};

// Worker thread (NO &Connection access):
let prompt_str: &str = &slot.prompt_bundle.prompt;
// Worker threads `shown_learning_ids` back to the main thread via SlotResult — the
// main thread cannot read SlotContext post-spawn because the closure consumed it.
let shown_ids: Vec<i64> = slot.prompt_bundle.shown_learning_ids.clone();
// ... runs Claude, returns:
SlotResult {
    slot_index: slot.slot_index,
    iteration_result, // includes conversation: Option<String>
    claim_succeeded: true,
    shown_learning_ids: shown_ids, // NEW field added by FEAT-002
}

// Main thread, post-join:
let shown_ids: &[i64] = &slot_result.shown_learning_ids;
process_iteration_output(&mut conn, &mut ctx, &mut IterationProcessingParams {
    task_id: &slot_result.task_id,
    output: &slot_result.iteration_result.output,
    claude_conversation: slot_result.iteration_result.conversation.as_deref(),
    outcome: &mut slot_result.iteration_result.outcome,
    shown_learning_ids: shown_ids,
    slot_index: Some(slot_result.slot_index),
    skip_git_completion_detection: true, // wave mode
    // ...
});
```
