# Claude Code Agent Instructions

You are an autonomous coding agent implementing **Auto-launch /review-loop after a successful loop / batch** for **task-mgr**.

## Problem Statement

After an autonomous loop completes ("all tasks complete" banner), the user currently has to manually open `claude` and type `/review-loop tasks/<prd>.md` to get a code review. This task list bakes that hand-off into `task-mgr` itself: on a clean loop exit (or end-of-batch for the last successful PRD meeting a task-count threshold), task-mgr spawns an interactive `claude` session with `/review-loop tasks/<prd>.md` as the initial prompt. The user lands directly in the review.

The default is on, gated to runs that completed **≥3 tasks**. CLI flags (`--auto-review` / `--no-auto-review`) and `.task-mgr/config.json` (`autoReview`, `autoReviewMinTasks`) override. For batch mode, ONE review fires at end-of-batch for the LAST successful PRD meeting the threshold — never per-PRD.

## Non-Negotiable Process (Read Every Iteration)

Before writing code:

1. **Internalize quality targets** — Read `qualityDimensions`; that's what "done well" means for THIS task.
2. **Plan edge-case handling** — For each `edgeCases` / `failureModes` entry on the task, decide how it'll be handled before coding.
3. **Pick an approach** — State assumptions in your head. Only for `estimatedEffort: "high"` or `modifiesBehavior: true` tasks, name the one alternative you rejected and why.

After writing code, the scoped quality gate is your critic — run it (Quality Checks § Per-iteration). Don't add a separate self-critique step; the linters, type-checker, and targeted tests catch more than a re-read does.

## Priority Philosophy

In order: **PLAN** (anticipate edge cases) → **PHASE 2 FOUNDATION** (~1 day now to save ~2+ weeks later — take it, we're pre-launch) → **FUNCTIONING CODE** (pragmatic, reliable) → **CORRECTNESS** (compiles, type-checks, scoped tests pass deterministically) → **CODE QUALITY** (clean, no warnings) → **POLISH** (docs, formatting).

Non-negotiables: tests drive implementation; satisfy every `qualityDimensions` entry; handle `Option`/`Result` explicitly (no `unwrap()` in production). For `estimatedEffort: "high"` or `modifiesBehavior: true` tasks, note the one alternative you rejected and why. For everything else, pick and go.

**Prohibited outcomes:**

- Tests that only assert 'no crash' or check type without verifying content
- Tests that mirror implementation internals (break when refactoring)
- Abstractions with only one concrete use
- Error messages that don't identify what went wrong
- Catch-all error handlers that swallow context
- Auto-review launch that masks a successful loop with its own failure — review launch errors must NEVER turn a passing loop into a failing one
- Spawning the interactive Claude with `--print` or `-p` (those force non-interactive) — must be a true TTY-attached session

- `auto_review::maybe_fire` not called from main.rs OR not called from batch.rs (must be both)

## CLAUDE.md Excerpts (only what applies to this change)

### Iteration pipeline (shared)

Sequential (`run_iteration`) and parallel-wave paths share a single post-Claude pipeline (`process_iteration_output` in `src/loop_engine/iteration_pipeline.rs`). Out of scope for the pipeline: wrapper-commit, external-git reconciliation, human-review trigger, rate-limit waits, pause-signal handling, slot merge resolution. **The auto-review hook is similarly out of scope for the pipeline** — it fires AFTER `run_loop` returns (in main.rs and at the END of `run_batch`), not per-iteration.

### Run-level config caching (restart required for mid-loop edits)

`ProjectConfig` and PRD-side overrides are loaded ONCE at `run_loop` startup and threaded through. Mid-loop edits to `.task-mgr/config.json` do NOT take effect — operators must restart the loop. **Same convention applies to FEAT-005**: read `ProjectConfig` once at the top of `run_batch` (not per-PRD), and re-read in `main.rs` once before the auto-review hook fires.

### Curate session cleanup workaround

Claude Code 2.1.110 writes an `ai-title` jsonl even with `--no-session-persistence`. Curate dedup/enrich pass a fixed `--session-id` and clean up synchronously after `child.wait()`. **Scope is narrow — loops and learning ingestion do NOT opt in; only curate sites do.** The new auto-review code likewise does NOT opt in (we want an INTERACTIVE session, not a transient `-p` print run, so the session-cleanup workaround doesn't apply).

### Worktree path

Worktrees live at `{repo}-worktrees/{branch}`. Each worktree has its own `.task-mgr/tasks.db` and its own copy of the PRD JSON. The loop reads the **worktree's** copy on each iteration. **For FEAT-005**: when reading `tasks/<prd>.md` for the auto-review markdown path, the file is in the worktree (where the user's branch lives), not the main repo. `auto_review::maybe_fire` should pass the JSON path that was used to start the loop (`run_config.prd_file`), which already resolves correctly.

### Loop CLI Cheat Sheet — relevant entries

- Never edit `tasks/*.json` directly — use the CLI and tags above
- Permission guard: loop iterations deny Edit/Write on `tasks/*.json` via `--disallowedTools`
- Spawn-fixup PRD targeting: when REVIEW-001 spawns FIX tasks, pass `--depended-on-by REVIEW-001` so the prefix is unambiguous

### ReviewLauncher trait shape (FEAT-003)

```rust
pub(crate) trait ReviewLauncher {
    fn launch(&self, md: &Path, worktree: Option<&Path>) -> std::io::Result<()>;
}

pub(crate) struct ProcessLauncher;
impl ReviewLauncher for ProcessLauncher {
    fn launch(&self, md: &Path, worktree: Option<&Path>) -> std::io::Result<()> {
        let claude_bin = std::env::var("CLAUDE_BINARY")
            .unwrap_or_else(|_| "claude".to_string());
        let prompt = format!("/review-loop {}", md.display());
        let mut cmd = std::process::Command::new(&claude_bin);
        cmd.arg(&prompt);
        if let Some(wt) = worktree {
            cmd.current_dir(wt);
        }
        // stdin/stdout/stderr inherit by default — interactive.
        // Env (ANTHROPIC_API_KEY etc.) inherits by default — required for auth.
        match cmd.status() {
            Ok(s) if s.success() => Ok(()),
            Ok(s) => {
                eprintln!("[hint] /review-loop session ended with status {s} \
                           — slash command may be unavailable or review reported issues");
                Ok(())
            }
            Err(e) => Err(e),
        }
    }
}

#[cfg(test)]
pub(crate) struct CapturingLauncher {
    pub(crate) calls: std::sync::Mutex<Vec<(PathBuf, Option<PathBuf>)>>,
}
#[cfg(test)]
impl ReviewLauncher for CapturingLauncher {
    fn launch(&self, md: &Path, worktree: Option<&Path>) -> std::io::Result<()> {
        self.calls.lock().unwrap().push((md.to_path_buf(), worktree.map(Path::to_path_buf)));
        Ok(())
    }
}
```

### Batch last-PRD selection (FEAT-005)

`PrdRunResult` already carries `tasks_completed: u32` and `worktree_path: Option<PathBuf>` (added in FEAT-002). The batch hook is a thin shell over a pure helper:

```rust
// In batch.rs (or auto_review.rs — pick whichever feels at home):
pub(crate) fn pick_review_target<'a>(
    results: &'a [PrdRunResult],
    decision: &Decision,
) -> Option<&'a PrdRunResult> {
    results.iter().rev().find(|r| {
        r.exit_code == 0
            && !r.skipped
            && !r.stopped
            && should_fire(decision, r.exit_code, /*was_stopped=*/ false, r.tasks_completed)
    })
}

// AFTER the per-PRD for loop ends AND AFTER worktree cleanup, BEFORE returning BatchResult:
let decision = auto_review::resolve_decision(&project_config, cli_force_on, cli_force_off);
if let Some(target) = auto_review::pick_review_target(&results, &decision) {
    // Build a synthetic LoopResult so maybe_fire can apply ALL its suppression
    // gates uniformly (TTY, worktree-missing, missing markdown).
    let synthetic = LoopResult {
        exit_code: 0,
        worktree_path: target.worktree_path.clone(),
        branch_name: target.branch_name.clone(),
        was_stopped: false,
        tasks_completed: target.tasks_completed,
    };
    auto_review::maybe_fire(
        &project_config,
        cli_force_on,
        cli_force_off,
        &synthetic,
        &target.prd_file,
        &launcher,
    );
}
```

**Critical**: do NOT call `launcher.launch` directly from batch.rs or main.rs. Always go through `maybe_fire`. This keeps the gate set (TTY check, worktree-existence check, missing-markdown check, decision check) centralized so the single-loop and batch paths cannot diverge.
