# Claude Code Agent Instructions

You are an autonomous coding agent implementing **Init Split + task-mgr enhance** for **task-mgr**.

## Problem Statement

`task-mgr init --from-json <prd>.json` today conflates two unrelated things: per-worktree scaffolding (`.task-mgr/`, migrations, default config) and PRD import (parse, prefix-assign, insert tasks). This PR splits them:

- `task-mgr init` (no PRD) — project-level only. Scaffolds `.task-mgr/`, runs migrations, writes default config, optionally fires the model picker. Idempotent.
- `task-mgr loop init <prd>` — PRD-level. Imports ONE PRD JSON.
- `task-mgr batch init <glob>...` — PRD-level. Imports many PRDs.
- `task-mgr init --from-json X` — preserved **indefinitely** as a deprecated shim that prints a stderr notice and dispatches to `loop init` (N=1) or `batch init` (N>1). The shim ALSO runs `init_project` first so the model picker still fires for today's operators.

Add a new command `task-mgr enhance` with subcommands `agents`, `show`, `strip` that writes/updates a marker-fenced (`<!-- TASK_MGR:BEGIN -->` / `<!-- TASK_MGR:END -->`) workflow block in CLAUDE.md AND AGENTS.md. Two profiles: `workflow` (task-mgr CLI cheat sheet + workflow patterns) and `full` (workflow + general LLM-coding guidelines). `task-mgr init --enhance` is a convenience flag combining both.

Loop and Batch top-level commands become parent-with-subcommand (`Init` / `Run`); the existing flat form (`task-mgr loop tasks/foo.json`) survives via `args_conflicts_with_subcommands` + a dispatch shim that synthesizes `LoopCommand::Run { ... }` and emits stderr deprecation.

## Priority Philosophy

In order: **PLAN** (anticipate edge cases) → **PHASE 2 FOUNDATION** (~1 day now to save ~2+ weeks later — take it, we're pre-launch) → **FUNCTIONING CODE** (pragmatic, reliable) → **CORRECTNESS** (compiles, type-checks, scoped tests pass deterministically) → **CODE QUALITY** (clean, no warnings) → **POLISH** (docs, formatting).

Non-negotiables: tests drive implementation; satisfy every `qualityDimensions` entry; handle `Option`/`Result` explicitly (no `unwrap()` in production). For `estimatedEffort: "high"` or `modifiesBehavior: true` tasks, note the one alternative you rejected and why. For everything else, pick and go.

**Prohibited outcomes:**

- Removing or changing the shim's row-level behavior — `init --from-json X --append --update-existing` MUST produce the same DB state as `loop init X --append --update-existing`. This is enforced by a golden-file equivalence test in FEAT-005.
- Touching any PRD JSON file from the new project-level `init_project` function — that side effect belongs to `loop init` / `batch init` only.
- Auto-firing the interactive model picker from `loop init` or `batch init` — picker is project-level config and fires only from `task-mgr init` (project-level) and from the deprecated `init --from-json X` shim path.
- Adding a `--force` flag to project-level `task-mgr init` — project-level reset is `rm -rf .task-mgr/`. Reject `--force` with empty `from_json` and exit 2 with a clear stderr message.
- Hand-editing `.claude/commands/tasks.md` — it is regenerated by `cargo run --bin gen-docs`.
- Removing pattern matches over `Commands::Loop { .. }` / `Commands::Batch { .. }` from non-dispatch sites and assuming the old fields persist — Learning [1069]: every exhaustive match must be updated to the new variant shape.
- Using `std::fs::write` in the new `enhance` module — all writes go through `marker_splice::write_atomic` (FEAT-001) to avoid torn writes.
- Tests that only assert "no crash" or check type without verifying content.
- Tests that mirror implementation internals (break when refactoring).

The Value-based pattern preserves unknown keys the user added (e.g. `customField: ...`); a strongly-typed `ProjectConfig::default().to_json()` would silently drop them.
